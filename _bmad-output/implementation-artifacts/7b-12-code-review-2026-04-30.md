### PATCH-4 - `substrate-frozen-paths-check.yml` parser is not robust enough for the frozen-range invariant

Evidence:
- `.github/workflows/substrate-frozen-paths-check.yml:54-66` uses `awk match($0, /.../, arr)`, which is a gawk extension and can fail on runners where `awk` is mawk or BusyBox awk.
- `.github/workflows/substrate-frozen-paths-check.yml:57-61` parses only the added-side hunk range. A deletion-only hunk touching frozen original lines can report an added range with count `0`, so the frozen old range can be missed.
- Current workspace has no diff to `app/marcus/orchestrator/dispatch_adapter.py:70-95`, so no immediate substrate violation exists.

Boundary check: the intended added-line boundary math for `+70`, `+95`, `+69`, and `+96` is correct when an added-side range exists. The defect is portability plus deletion-only old-range coverage.

Required remediation: replace the awk block with POSIX-safe parsing or a small Python parser, parse both `-old_start,old_count` and `+new_start,new_count`, and add a lightweight boundary fixture/check for `+70`, `+95`, `+69`, `+96`, `-70,1 +69,0`, and `-95,1 +94,0`.
