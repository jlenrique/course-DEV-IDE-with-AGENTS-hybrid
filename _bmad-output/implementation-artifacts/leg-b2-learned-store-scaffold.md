## Why (product value / guardrail)
Stand up the **write-target + enforcement machinery** for learned Gamma-dependency rules so that, once Leg-E's live audit produces real observations, a promoted rule can be validated offline and can't silently vanish or mutate. B2 is a **scaffold**: it builds the schema, the append-only observations ledger (the Leg-E write-target), the identity-manifest pin, and the validator's interpretation of active learned rules — proven by a synthetic fixture. It ships **ZERO active learned rules** (manifest empty by design). PRODUCT-VALID (the enforcement half of the copacetic-quality audit); NOT proofing-driven.

## Ratified architecture (Leg-B green-light — binding)

Promoted RULES → CD-authored `learned_dependencies` YAML block via the envelope ONLY, provenance-cited, validator-gated. (The BMAD memory-sidecar *projection* — a read-optimized rebuildable view — is DEFERRED; the ledger is the sole SSOT.)
- **Promotion authority (Dan):** automation PROPOSES a candidate into the ledger; **CD RATIFIES-and-authors** the promoted rule via the envelope. Automation never holds the pen. **DEFERRED in B2** (no promotion path built) — B2 ships the schema + ledger + pin + interpretation; the automation-proposes/CD-ratifies live path + the live envelope-authoring ceremony land post-Leg-E.
- **Identity-manifest pin (Murat):** `state/config/gamma-learned-rules.lock` = sorted manifest of `{rule_id, provenance_run_ref, fixture_path, predicate_hash}`. The validator asserts the live `learned_dependencies` rule-id set **⊇ manifest** (superset — append-only, never equality-that-drops); per-entry **predicate_hash** match (no silent in-place mutation); count = secondary tripwire. A drop OR a swap goes RED naming the offending id.
- **Anti-vacuous-green guard (Murat, BLOCKING):** the manifest **starts EMPTY by design.** The 3 candidates are seeded as `status: candidate` **OBSERVATIONS in the ledger, NOT** promoted rules and NOT in the manifest. Every manifest entry (i.e. every ACTIVE learned rule) must resolve to a committed RED-proven birthing fixture (without-rule fails / with-rule passes). A non-empty `learned_dependencies` active set with no manifest row or no fixture = RED. An empty active set is honest ONLY when the manifest is explicitly declared empty.

## The build (scaffold)
1. **`learned_dependencies` schema** — a new module `app/specialists/gary/learned_dependencies.py` defining the declarative entry shape: `{rule_id, antecedent{field, predicate}, consequent{field, expected}, severity(fail|warn), provenance{observation_ids[], birthing_run_ref, fixture_path, promoted_at, cd_commit}, status(active|candidate)}`. A pure interpreter `apply_learned_rules(record_or_resolved, rules) -> (errors, warnings)` that evaluates active rules identically to the hardcoded ones (no free-form logic — declarative predicate only).
2. **Observations ledger** — a file-based append-only store `state/config/gamma-learned-observations.jsonl` + a helper `append_observation(...)` in the same module that is **digest-keyed idempotent** (an observation carries `{observation_id, observed_at, output_digest, source_component, behavior}`; a write whose `output_digest` collides is a no-op → structurally append-only, no live fetch needed). Seed the **3 candidates as `status: candidate` observations** (model UI-name≠API-string churn; **burst-throttle 401≠429**; account parallel-task cap) — each with a captured birthing-evidence reference. These are OBSERVATIONS, not learned rules.
3. **Identity-manifest pin** — `state/config/gamma-learned-rules.lock` (committed, **empty manifest** at ship) + a check in `validate_style_guides_full`: parse `learned_dependencies` (active entries), assert id-set ⊇ manifest, predicate_hash match per entry, and every active entry resolves to an existing `fixture_path`. Missing manifest row / hash mismatch / missing fixture → error `gamma.learned.pin-violation`.
4. **Validator interpretation** — apply active learned rules (from the YAML block, gated by the pin) per-record via the interpreter; ERROR/WARN routed to the existing channels.
5. **Anti-vacuous guard** — a test that the SHIPPED config (empty `learned_dependencies`, empty manifest) passes cleanly, AND that a synthetic active learned rule WITHOUT a manifest row / fixture goes RED.

## Acceptance criteria (RED-first; dual-gate)
- **AC#1 [schema + interpreter]** `apply_learned_rules` evaluates a declarative active rule (antecedent⇒consequent) and emits ERROR or WARN per `severity`; a synthetic fixture rule fires on a violating record, silent on a good one.
- **AC#2 [append-only ledger, digest-idempotent]** `append_observation` appends a new observation; a re-append with the same `output_digest` is a no-op (idempotent); the ledger file is never rewritten/shrunk. Test the idempotency + append-only invariant.
- **AC#3 [3 candidates seeded as OBSERVATIONS]** the ledger contains the 3 candidates with `status: candidate` + birthing-evidence refs; NONE appear in `learned_dependencies` (active rules) and NONE in the manifest. Assert they do not enforce.
- **AC#4 [identity-manifest pin — superset]** with the shipped empty manifest + empty `learned_dependencies`, the pin passes. A synthetic `learned_dependencies` active entry NOT in the manifest → `gamma.learned.pin-violation`. A manifest row whose `predicate_hash` ≠ the live rule's hash → RED (in-place-mutation guard). A manifest row is a SUPERSET check — adding a new active rule+manifest row passes; removing/renaming a manifested rule → RED.
- **AC#5 [anti-vacuous, BLOCKING]** an active learned rule WITHOUT a committed RED-proven birthing fixture (without-rule fails / with-rule passes) → RED. The empty-manifest ship state is honest (explicitly declared empty). A non-empty active set with no fixtures cannot pass.
- **AC#6 [hermetic]** no network, no DB — `apply_learned_rules` + the pin + the ledger helper are pure file/in-memory; a test asserts the default `validate_style_guides_full` path (now including the pin) makes no outbound call and touches no psycopg.
- **AC#7 [🔒 back-compat + seeds clean]** the 3 shipped styleguide seeds + B1's rules still validate 0 err/0 warn; `validate_style_guides`/`_full` back-compat unchanged for callers; B1's 25 tests stay green.
- **AC#8 [honesty disclaimer — Dan, BINDING]** the story + a module docstring explicitly state: B2 validates the CD-authoring path **by fixture, NOT exercised live**; the live envelope-authoring ceremony + the automation-proposes/CD-ratifies promotion path are DEFERRED to `styleguide-cd-envelope-authoring-ceremony` (post-Leg-E). No over-claim.
