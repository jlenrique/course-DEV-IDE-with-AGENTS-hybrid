## Voice-selection 3-way verb-payload consistency note

The `Section11OperatorVerdict._enforce_verb_payload_consistency()` `model_validator(mode="after")` correctly enforces required-iff + forbids semantics for **all three verbs** (select / edit / reject):

- `verb="select"` → requires `select_payload: VoiceSelectionPayload`; forbids `edit_payload` and `reject_reason`.
- `verb="edit"` → requires `edit_payload: VoiceSelectionEditPayload`; forbids `select_payload` and `reject_reason`.
- `verb="reject"` → requires `reject_reason`; forbids `select_payload` and `edit_payload`.

This is structurally **richer than §02A canonical**, where `approve` is null-payload (no fields required) and only `edit`/`reject` carry mandatory payloads (2-way enforcement). §11's 3-way enforcement is the correct pattern for voice-selection: the `select` verb carries operator-chosen `selected_voice_id` as REQUIRED state (not null-payload), so it must enforce a positive `select_payload` constraint. Codex correctly diverged from §02A's 2-way pattern because the verb semantics differ — `select` is a positive selection action, not an approval rubber-stamp.

The verb literal `Literal["select", "edit", "reject"]` per FR-7c-18 voice-picker semantics — verified NOT mistakenly authored as `["approve", "edit", "reject"]` (the §02A pattern). Pattern-parity correct.
