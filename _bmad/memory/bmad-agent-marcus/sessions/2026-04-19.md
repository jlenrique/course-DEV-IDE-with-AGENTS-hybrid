### Prompt Pack §01 Sequencing Glitch (noted by operator)

**Issue:** Pack §01 (Activation + Preflight) is listed as Prompt 1, but it requires `RUN_ID` and `BUNDLE_PATH` as concrete template values to execute its required commands (emit_preflight_receipt, app_session_readiness) and write `preflight-results.json` to the bundle. Those values are outputs of PR-RC (run constants authoring), which the pack treats as a pre-run setup step rather than a numbered prompt.

**Effect:** §01 cannot be executed first in a clean greenfield run without first establishing run constants. The pack's implied sequence (PR-RC → §01) is correct operationally but not explicit in the pack's own numbering. A fresh operator following only the numbered prompts would hit this dependency before having a RUN_ID.

**Operator's characterization:** "a glitch in our approach" — not a blocker, but worth flagging for pack maintenance.

**Recommended fix (for pack v4.3 or next pack update):**
- Add a §00 or a pre-numbered "Run Constants" step before §01, or
- Make §01 itself surface the PR-RC questions inline before running the required commands, or
- Document the PR-RC prerequisite explicitly at the top of §01

**Status:** Noted. Not remediated in this session. Carry forward to pack governance.

### §01 Preflight Double-Run (pack improvement candidate)

**Observation:** Preflight ran twice this session — once as PR-PF (session-level ecosystem check) and once as §01's `emit_preflight_receipt` (bundle-anchored artifact write). Both are correct and serve different purposes: PR-PF confirms tool readiness; §01 writes the canonical receipt the pipeline validator checks at every downstream gate.

**Recommended fix (pack v4.3):** If PR-PF already ran and passed in the current session, §01 should be allowed to write the receipt from the cached result rather than re-pinging all APIs. Add a `--use-cached-preflight` flag or session-level receipt handoff to `emit_preflight_receipt`.

**Status:** Noted. Not remediated. Carry forward to pack governance.
